Website URL  : https://web-mall-sand.vercel.app

"WebMall is a full-stack e-commerce website built with Next.js and TypeScript — so the same language runs on both the frontend and backend. 
For the UI, I used React with Tailwind CSS and shadcn/ui components. The database is PostgreSQL hosted on Supabase, and I interact with it using Prisma ORM. 
Authentication is also handled through Supabase Auth. The whole thing is deployed on Vercel with automatic CI/CD. 
Since Next.js handles both the server and the client, I didn't need a separate backend — API routes, server components, and the React UI all live in one codebase."

"why Next.js?" — say "because it gives me SSR, API routes, and image optimization out of the box, so I didn't need to set up a separate Express or Django server."

  LANGUAGES AT A GLANCE
================================================================================

  Backend Language  : TypeScript / JavaScript (Node.js runtime)
                      - All server-side API routes, database queries, and
                        server components are written in TypeScript (.ts)
                      - Runs on Node.js via the Next.js server layer
                      - Prisma ORM queries, auth logic, and business rules
                        all execute server-side in TypeScript

  Frontend Language : TypeScript + TSX (React / Next.js)
                      - All UI components are written in TypeScript with JSX
                        syntax (.tsx files)
                      - React 18 is the UI rendering library
                      - Client components use React hooks and context API
                      - Styled with Tailwind CSS utility classes

  Why the Same Language for Both?
                      Next.js is a full-stack framework built entirely around
                      JavaScript/TypeScript, which means both the server and
                      the browser naturally share the same language. Here is
                      why this is a deliberate and practical choice:

                      1. UNIFIED CODEBASE
                         A single language across the entire project means one
                         team, one repo, and no context-switching between a
                         separate backend language (e.g. Python, Java, PHP)
                         and a frontend language. All code lives together.

                      2. SHARED TYPE DEFINITIONS
                         TypeScript interfaces and types (e.g. Product, Order,
                         User) are defined once and reused by both API route
                         handlers (server) and React components (client),
                         eliminating mismatches between what the API returns
                         and what the UI expects.

                      3. NEXT.JS FULL-STACK MODEL
                         Next.js deliberately blurs the server/client boundary.
                         Server Components fetch data on the server; Client
                         Components render in the browser. Both are written in
                         the same .tsx syntax — the framework decides at build
                         time what runs where.

                      4. CODE REUSE
                         Validation schemas (Zod), utility functions, constants,
                         and data models are written once and imported by both
                         API routes and UI components without any translation
                         layer or duplication.

                      5. DEVELOPER EXPERIENCE
                         One language means one linter (ESLint), one type
                         checker (TypeScript), one build tool (Next.js/SWC),
                         and one package manager (npm). This drastically
                         reduces tooling complexity.

                      6. ECOSYSTEM
                         The JavaScript/TypeScript npm ecosystem is the largest
                         in the world. Every library needed — from Prisma (DB)
                         to Framer Motion (animation) to Zod (validation) — is
                         available as a TypeScript-first package, with full
                         type support out of the box.

--------------------------------------------------------------------------------
  1. CORE FRAMEWORK & RUNTIME
--------------------------------------------------------------------------------

  Framework     : Next.js (v16.x)
                  - App Router architecture (app/ directory)
                  - Server-Side Rendering (SSR) + Static Generation (SSG)
                  - Turbopack enabled for faster local development
                  - API Routes via app/api/ directory
                  - Route-level loading.tsx and error.tsx support
                  - Image optimization with next/image (WebP + AVIF formats)

  Language      : TypeScript (v5.x)
                  - Strict type checking
                  - Compiled with tsc

  Runtime       : Node.js
                  - Server-side Prisma queries run in Next.js server components

--------------------------------------------------------------------------------
  3. BACKEND - API & SERVER
--------------------------------------------------------------------------------

  API Layer     : Next.js API Routes (app/api/)
                  - RESTful endpoints co-located with the app
                  - Server Actions for form mutations (Next.js 14+ pattern)

  Key API Routes:
    /api/products/        - Product CRUD
    /api/categories/      - Category management
    /api/cart/            - Cart operations
    /api/wishlist/        - Wishlist operations
    /api/orders/          - Order creation & management
    /api/admin/           - Admin-only endpoints (dashboard, users, analytics)
    /api/coupons/         - Coupon validation & usage tracking
    /api/reviews/         - Product reviews
    /api/messages/        - Contact messages (user <-> admin)
    /api/settings/        - Site config (store name, shipping rates, socials)
    /api/upload/          - Image upload to Supabase Storage

--------------------------------------------------------------------------------
  4. DATABASE & ORM
--------------------------------------------------------------------------------

  Database      : PostgreSQL
                  - Hosted on Supabase (managed Postgres)
                  - UUID primary keys (uuid_generate_v4())
                  - Snake_case column naming convention

  ORM           : Prisma (v6.x)
                  - prisma-client-js generator
                  - multiSchema preview feature enabled
                  - Schema at prisma/schema.prisma
                  - CamelCase model names mapped to snake_case DB tables

  Database Models (Tables):
    users              - Customer accounts, roles (customer/admin)
    categories         - Product top-level categories
    subcategories      - Nested subcategories under each category
    products           - Product listings (price, stock, status)
    product_images     - Multiple images per product (with position ordering)
    product_variants   - SKU-based variants (color, size, etc.) with attributes JSON
    carts              - One cart per user
    cart_items         - Items in a cart (product + optional variant)
    wishlists          - One wishlist per user
    wishlist_items     - Items in a wishlist
    orders             - Customer orders (shipping address JSON, payment method)
    order_items        - Line items per order
    reviews            - Star ratings + comments per product per user
    contact_messages   - User contact form submissions with admin reply support
    site_settings      - Single-row store config (shipping rates, social URLs, nav)
    hero_banners       - Homepage rotating hero banners with CTA config
    coupons            - Discount codes (fixed/percent, usage limits, expiry)
    coupon_usage       - Per-user coupon redemption tracking
    inventory_items    - Internal stock tracking

--------------------------------------------------------------------------------
  11. ARCHITECTURE OVERVIEW
--------------------------------------------------------------------------------

  Pattern       : Hybrid SSR + CSR (Next.js App Router)

    - Server Components  : Data fetching pages (products, home, categories)
                           No client-side bundle shipped for pure data pages
    - Client Components  : Interactive UIs (cart, wishlist, auth forms, admin)
                           Marked with 'use client' directive
    - API Routes         : Stateless REST handlers for mutations and reads
                           that need auth checks or DB access from client

  Folder Structure:
    app/                 - Pages, layouts, API routes (Next.js App Router)
    components/          - Shared React components
      ui/                - shadcn/ui primitive components
      layout/            - Header, Footer, ScrollbarHandler
      products/          - ProductCard, product-related UI
      home/              - InstagramShowcase, home-specific widgets
      admin/             - Admin dashboard components
      auth/              - Auth overlay, verification components
    contexts/            - React Context providers
    hooks/               - Custom React hooks
    lib/                 - Prisma client, utility functions, constants
    prisma/              - Schema and migrations
    public/              - Static assets (logo, OG image)
    types/               - Shared TypeScript type definitions
